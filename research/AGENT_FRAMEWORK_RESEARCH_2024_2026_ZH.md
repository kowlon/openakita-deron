## 1. 主要开源 Agent 框架

### 1.1 LangGraph / LangChain Agents

| 维度 | 说明 |
|---|---|
| **仓库** | [langchain-ai/langgraph](https://github.com/langchain-ai/langgraph) |
| **版本** | LangGraph 1.0 alpha（2025 年 9 月） |
| **许可证** | MIT |

**核心架构 / 设计理念**

LangGraph 是一个建立在**有向图抽象**之上的底层编排框架。Agent 被建模为有状态图，节点代表计算步骤（LLM 调用、工具执行、分支逻辑），边代表状态转移。其设计理念强调**可控性与持久性高于易用性**——以最小抽象让开发者获得最大控制力。

该框架采用生产优先路线，聚焦三个核心挑战：
1. **延迟管理**——应对从数秒到数小时的多次 LLM 调用
2. **失败恢复**——基于检查点重试，避免昂贵的重新执行
3. **非确定性 AI**——支持审批、测试与可复现性

**关键创新**
- **持久化检查点**：每次状态转移都做检查点，支持时间回溯调试、失败恢复与人工审批流
- **持久执行**：长任务可跨进程重启继续运行，状态可持久化与恢复
- **流式优先**：内建对深层图执行的 token、状态与事件流式输出
- **子图**：将 Agent 行为模块化为可嵌套图
- **中断**：可编程暂停/恢复，用于人工审批、外部输入或条件逻辑

**处理方式**
- **工具**：图中的一等节点；以图转移的方式调用，自动重试与错误处理
- **记忆**：内建持久层与检查点；短期（图运行态）与长期（跨运行存储）
- **规划**：隐含于图拓扑；也支持由 LLM 路由节点驱动的动态规划
- **多智能体**：子图实现层级化模式，Agent 作为节点嵌入更大的编排图
- **错误恢复**：从上次成功检查点重试，无需重复昂贵 LLM 调用

**优势**
- 开发者控制力最大，无隐藏抽象
- 大规模生产验证（Klarna、LinkedIn、Uber、Elastic）
- 通过时间回溯与状态检查具备优秀调试体验
- 生态完善（LangSmith 可观测性、LangGraph Platform 部署）
- 基准中比 CrewAI 快 2.2 倍，依赖状态增量传递带来最佳 token 效率

**劣势**
- 学习曲线陡峭
- 需要前期图式设计投入
- 对简单单轮工具调用过于重型
- 存在与 LangSmith/LangGraph Platform 的生态锁定风险

### 1.2 CrewAI

| 维度 | 说明 |
|---|---|
| **仓库** | [crewAIInc/crewAI](https://github.com/crewAIInc/crewAI) |
| **许可证** | MIT |

**核心架构 / 设计理念**

CrewAI 采用受人类团队协作启发的**角色驱动多智能体范式**。核心抽象将 Agent 视为拥有角色、目标与背景的团队成员。Crew（团队）通过可配置流程执行任务，理念是“AI Agent 作为协作团队成员”。

**核心原语**：
- **Agents**：具备角色定义、工具、记忆、知识与结构化（Pydantic）输出的自治单元
- **Tasks**：分配给 Agent 的离散任务与期望输出
- **Crews**：协作 Agent 的集合
- **Flows**：更高层控制（start/listen/router）、状态管理与长流程编排
- **Processes**：顺序、层级或混合执行策略

**关键创新**
- **角色化 Agent 设计**：角色/目标/背景设定，映射真实组织结构
- **流程灵活性**：顺序（流水线）、层级（经理委派）与混合
- **内建护栏与回调**：任务级校验与人工介入触发
- **企业集成**：Gmail、Slack、Salesforce 等原生连接器

**处理方式**
- **工具**：Agent 配备工具集，支持 MCP 服务扩展
- **记忆**：内建知识库式记忆，支持持久化与召回
- **规划**：流程类型定义规划策略
- **多智能体**：以 Crew 为拓扑核心，显式角色分配
- **错误恢复**：任务级回调与护栏，边缘情况人工介入

**优势**
- API 直观，原型速度快
- 生态集成丰富（Langfuse、Datadog、Braintrust 可观测性）
- 角色化设计契合业务流程
- 文档与社区良好

**劣势**
- 基准中“Agent 到工具”的间隙开销 >5 秒
- token 消耗高于图式方法（保留完整上下文）
- 控制粒度不如 LangGraph
- 抽象可能掩盖规模化性能问题

### 1.3 AutoGen（Microsoft）

| 维度 | 说明 |
|---|---|
| **仓库** | [microsoft/autogen](https://github.com/microsoft/autogen) |
| **版本** | v0.4（2025 年 1 月——全面重写） |
| **许可证** | MIT |

**核心架构 / 设计理念**

AutoGen v0.4 基于**演员模型（actor model）**，支持分布式、可扩展、事件驱动的 Agent 系统。采用**分层架构**：
- **核心层**：事件驱动消息、Agent 运行时、分布式执行
- **AgentChat 层**：高层多 Agent 对话模式（保留 v0.2 API）

理念强调**可组合性、灵活性与可扩展性**——不同框架/语言的 Agent 可集成。

**关键创新**
- **演员模型**：每个 Agent 拥有独立邮箱，实现跨进程/跨机器分布式执行
- **框架无关组合**：整合不同框架或语言的 Agent
- **Magentic-One**：通用多 Agent 团队（浏览、文件、编码）
- **AutoGen Studio**：低代码可视化构建与测试
- **双工作流**：既支持确定性流程，也支持事件驱动分布式流程

**处理方式**
- **工具**：函数型定义，可共享或独占
- **记忆**：AgentChat 层内建记忆管理，支持序列化与持久化
- **规划**：结构化预定义模式或事件驱动的涌现式规划
- **多智能体**：群聊、双 Agent、Magentic-One 层级团队
- **错误恢复**：事件驱动通信集中化，状态序列化支持检查点

**优势**
- 跨进程/跨机器分布式执行
- 最灵活的多 Agent 对话模式
- 微软研究支持强
- AutoGen Studio 可视化开发
- 支持 Python 3.10+ 与 .NET SDK

**劣势**
- v0.4 重写导致生态迁移成本
- token 效率挑战与 LangChain 类似（保留完整历史）
- 对简单场景复杂度过高
- AutoGen 与 AG2 分叉导致社区碎片化

### 1.4 OpenAI Agents SDK（Swarm 继任者）

| 维度 | 说明 |
|---|---|
| **仓库** | [openai/openai-agents-python](https://github.com/openai/openai-agents-python) |
| **前身** | Swarm（实验性，已弃用） |
| **许可证** | MIT |

**核心架构 / 设计理念**

Agents SDK 是 Swarm 的生产级继任者，保留轻量极简哲学并加入生产功能，遵循两条原则：
1. 足够生产可用，但保持少量原语便于学习
2. 开箱即用，同时允许定制

**核心原语**：
- **Agents**：带指令与工具的 LLM
- **Handoffs**：Agent 委派到其他 Agent 的机制
- **Guardrails**：输入/输出验证

**关键创新**
- **Handoff 模式**：自然的 Agent 之间上下文交接
- **内建 Agent 循环**：自动工具调用直到完成任务
- **Python 优先编排**：使用原生 Python 语言特性而非 DSL
- **Sessions**：跨运行持久化上下文
- **Realtime Agents**：语音 Agent 与中断检测
- **模型提供方无关**：支持 100+ LLM

**处理方式**
- **工具**：函数型定义，自动 JSON Schema；支持 MCP 集成
- **记忆**：Session 提供持久上下文与历史管理
- **规划**：Agent 循环处理工具序列；handoff 用于任务委派
- **多智能体**：基于 handoff 的网络式委派
- **错误恢复**：工具重试与前后置校验护栏

**优势**
- API 极简、学习成本低
- 生产可用，内建追踪与监控
- Handoff 模式直观强大
- 提供方无关
- 轻量无重型依赖

**劣势**
- 编排能力不如 LangGraph/AutoGen
- Handoff 不适合所有多 Agent 拓扑
- 新框架，企业级验证有限
- 内建记忆能力有限（仅 session）

### 1.5 Anthropic 的 Agent 模式

| 维度 | 说明 |
|---|---|
| **参考** | [Building Effective Agents](https://www.anthropic.com/research/building-effective-agents) |
| **定位** | 设计模式，而非框架 |

**核心架构 / 设计理念**

Anthropic 明确**不提供框架**，而倡导使用**简单、可组合的模式**直接基于 LLM API 实现。其核心观点：多数成功 Agent 实现使用简单模式，多数任务并不需要完全自主 Agent。

关键区分：**工作流**（预定义路径）与 **Agent**（LLM 动态引导流程）。

**五类工作流模式**：
1. **提示链**：顺序 LLM 调用，每步质量门控
2. **路由**：分类输入并导向专门处理器
3. **并行化**：同时执行多个 LLM 调用（分段或投票）
4. **编排者-工作者**：中央 LLM 拆解任务并分派
5. **评估-优化**：生成与评估循环迭代

**何时使用 Agent**：仅当需要灵活性与模型驱动决策时；任务明确时用工作流。

**关键创新**
- **简化原则**：“先用直接 API 调用，大多数模式几行代码可实现”
- **上下文工程**（2025 进化）：管理完整输入栈，而非只做 prompt
  1. 系统指令（角色、行为、约束）
  2. 长期记忆（偏好、历史决策）
  3. 检索文档（RAG）
  4. 工具定义（函数与 schema）
  5. 对话历史
  6. 当前任务
- **反模式**：巨石型 Agent、过度规划、缺失可观测性

**优势**
- 实用且经过真实案例验证（Coinbase、Intercom、Thomson Reuters）
- 无厂商锁定
- 倡导简单与渐进复杂化
- 有助于选择合适模式的思维模型

**劣势**
- 非框架，需要更多工程实现
- 缺乏现成持久化、追踪与部署工具
- 模式泛化，细节需自行实现
- 对复杂多 Agent 场景指导不足

### 1.6 Google ADK（Agent Development Kit）

| 维度 | 说明 |
|---|---|
| **仓库** | [google/adk-python](https://github.com/google/adk-python) |
| **语言** | Python、TypeScript、Go、Java |
| **许可证** | Apache 2.0 |

**核心架构 / 设计理念**

Google ADK 旨在让**开发 Agent 像开发软件**。它是模块化开源框架，清晰区分 LLM 驱动 Agent（推理）与工作流 Agent（确定性控制）。

**核心组件**：
- **Agents**：`LlmAgent`（推理）+ 工作流 Agent（`SequentialAgent`、`ParallelAgent`、`LoopAgent`）
- **Tools**：自定义函数、Agent-as-Tool、代码执行、外部数据源
- **Events**：通信单元（消息、回复、工具使用）
- **Runner**：执行引擎与编排
- **Session & State**：对话上下文、历史与工作记忆
- **Memory**：跨会话长期记忆
- **Artifacts**：文件与二进制数据管理

**关键创新**
- **工作流 Agent 原语**：内建顺序/并行/循环 Agent
- **Agent-as-Tool**：将完整 Agent 作为工具调用
- **多语言支持**：Python、TS、Go、Java
- **Vertex AI 集成**：无缝部署到 Google 云 Agent Engine
- **开发工具**：Web UI（`adk web`）、CLI（`adk run`）、API Server（`adk api_server`）

**处理方式**
- **工具**：丰富生态，支持代码沙箱与外部数据连接
- **记忆**：短期 session 状态 + 长期记忆服务
- **规划**：工作流结构化规划 + LLM 动态路由
- **多智能体**：层级组合，专用 Agent 由上级协调
- **错误恢复**：Loop Agent 重试；事件驱动错误传播

**优势**
- 最完整的 Agent 开发体验（IDE/CLI/Web/云部署）
- 语言支持最广（4 种）
- 推理与确定性逻辑分离清晰
- Vertex AI 企业部署路径强
- 背靠 Google 76 页白皮书设计原则

**劣势**
- 2025 年新框架，社区较小
- 与 Google 云生态耦合更紧
- 社区工具不如 LangChain 生态
- 文档仍在完善中

### 1.8 MetaGPT

| 维度 | 说明 |
|---|---|
| **仓库** | [geekan/MetaGPT](https://github.com/geekan/MetaGPT) |
| **版本** | 0.8.2（2025 年 3 月） |
| **许可证** | MIT |

**核心架构 / 设计理念**

MetaGPT 将**元编程与标准作业流程（SOP）**应用于多 Agent 协作，模拟人类软件开发流程：产品、架构、工程、测试等角色按 SOP 产出可验证工件。

**关键原则**：
- **流水线范式**：复杂任务拆解为流水线子任务
- **SOP 驱动协同**：人类流程编码为提示序列
- **模块化产出**：每个角色输出可验证工件

**关键创新**
- **SOP 编码**：将组织流程转化为协作协议
- **级联幻觉缓解**：中间工件验证阻止错误传播
- **软件工程特化**：代码生成质量高于 AutoGPT 与 LangChain 的协作基准

**处理方式**
- **工具**：按角色配置工具（研究用检索、工程用执行等）
- **记忆**：共享工作区进行结构化工件交换
- **规划**：SOP 定义计划，角色明确输入/输出
- **多智能体**：基于角色的流水线与清晰交接
- **错误恢复**：中间验证与 QA 角色终检

**优势**
- 适合软件工程流程
- SOP 降低错误级联
- 协作产出质量高
- 执行链可预测

**劣势**
- 非软件任务灵活性差
- SOP 刚性限制动态调整
- 社区规模小于 LangGraph/AutoGen
- Python 3.9–3.11 限制兼容

## 2. 关键学术论文与概念

### 2.1 ReAct：Reasoning + Acting（Yao et al., 2022）

| 维度 | 说明 |
|---|---|
| **论文** | “ReAct: Synergizing Reasoning and Acting in Language Models” |
| **会议** | ICLR 2023 |
| **作者** | Shunyu Yao, Jeffrey Zhao, Dian Yu, Nan Du, Izhak Shafran, Karthik Narasimhan, Yuan Cao |

**核心概念**

ReAct 将**推理轨迹**（思维链）与**任务动作**在同一生成循环中交替：
- **Thought**：内部推理，跟踪计划、更新信念、处理异常
- **Action**：外部交互（API、检索、命令执行）
- **Observation**：接收环境反馈

**关键创新**：推理与行动不再分离，推理指导动作，动作结果反过来影响推理。

**结果**：在 HotpotQA、Fever、ALFWorld、WebShop 等任务表现出色，成为几乎所有现代 Agent 框架默认模式。

**演化**：ReflAct（2025，EMNLP）从“计划下一步”转向“持续反思目标对齐”，在 ALFWorld 上达到 93.3% 成功率（+27.7%）。

**影响**：ReAct 已成为事实上的标准 Agent 循环模式。

### 2.2 Reflexion（Shinn et al., 2023）

| 维度 | 说明 |
|---|---|
| **论文** | “Reflexion: Language Agents with Verbal Reinforcement Learning” |
| **会议** | NeurIPS 2023 |
| **作者** | Noah Shinn, Federico Cassano, Ashwin Gopinath, Karthik Narasimhan, Shunyu Yao |

**核心概念**

Reflexion 让 Agent 通过**语言反思**从失败中学习，而非参数更新。失败后生成自然语言批评，存入情节记忆；后续尝试利用反思指导决策。

**架构**：
1. **Actor**：生成动作
2. **Evaluator**：评分轨迹
3. **Self-Reflection**：生成失败分析
4. **Episodic Memory**：积累反思

**关键创新**：用语言反思替代梯度更新，让 Agent 通过阅读自身反思改进。

**结果**：HumanEval pass@1 达到 91%（超过 GPT-4 的 80%）。

**影响**：形成“尝试 → 失败 → 反思 → 重试”的标准模式。

### 2.3 Plan-and-Solve（Wang et al., 2023）

| 维度 | 说明 |
|---|---|
| **论文** | “Plan-and-Solve Prompting: Improving Zero-Shot Chain-of-Thought Reasoning” |
| **会议** | ACL 2023 |

**核心概念**

Plan-and-Solve（PS）解决零样本思维链的三类问题：计算错误、遗漏步骤与语义误解。两阶段：
1. **Plan**：拆解子任务
2. **Solve**：按计划执行

PS+ 通过更详细指令降低计算错误。

**关键创新**：显式分离“规划”与“执行”。

**演化（2024）**：
- **基于 QDMR 的 PS**：用子问题 DAG 表达逻辑依赖
- **Planning Tokens**：推理步骤开头生成规划 token
- **PEARL**：把问题拆成可执行步骤（如 SUMMARIZE、FIND_RELATION）

**影响**：成为编排者-工作者与多步 Agent 的标准模式。

### 2.4 Tree of Thoughts（Yao et al., 2023）

| 维度 | 说明 |
|---|---|
| **论文** | “Tree of Thoughts: Deliberate Problem Solving with Large Language Models” |
| **会议** | NeurIPS 2023 |
| **作者** | Shunyu Yao, Dian Yu, Jeffrey Zhao, Izhak Shafran, Thomas L. Griffiths, Yuan Cao, Karthik Narasimhan |

**核心概念**

Tree of Thoughts（ToT）将思维链扩展为**多路径探索**。受双过程理论启发，引入“系统 2”式的刻意思考：
- **思路分解**：拆成中间步骤
- **思路生成**：每步生成多个候选
- **状态评估**：自评最优分支
- **搜索算法**：BFS/DFS 回溯

**关键创新**：LLM 可自评与回溯，实现策略性前瞻探索。

**结果**：Game of 24 成功率 74%（思维链仅 4%）。

**影响**：奠定树搜索式规划（LATS、Monte Carlo）的基础。

### 2.5 Agent-as-a-Judge（Zhuge et al., 2024）

| 维度 | 说明 |
|---|---|
| **论文** | “Agent-as-a-Judge: Evaluate Agents with Agents” |
| **会议** | ICML 2025 |
| **仓库** | [microsoft/AgentAsJudge](https://github.com/microsoft/AgentAsJudge) |

**核心概念**

将 LLM-as-a-Judge 扩展为**Agent 评估 Agent**，不仅评估最终输出，还评估**全过程**（推理、工具使用与决策质量）。

**关键创新**：
- **过程级评估**：审查中间推理与工具使用
- **DevAI 基准**：55 个真实开发任务、365 层级要求、丰富人工标注
- **Agent 化评估**：评审 Agent 可用工具、访问文件、运行代码验证

**结果**：显著优于传统 LLM-as-a-Judge，可靠性接近人工评估。

**影响**：确立“评估 Agent 需要 Agent”的范式。

### 2.6 LATS：Language Agent Tree Search（Zhou et al., 2024）

| 维度 | 说明 |
|---|---|
| **论文** | “Language Agent Tree Search Unifies Reasoning, Acting, and Planning” |
| **会议** | ICML 2024 |

**核心概念**

LATS 将**蒙特卡洛树搜索（MCTS）与语言模型 Agent**结合，把 LLM 同时作为：
1. **Agent**：生成动作
2. **价值函数**：评估状态
3. **优化器**：选择探索分支

**架构**：
- 选择 → 扩展 → 评估 → 仿真 → 回传（MCTS 循环）
- 引入自反思作为上下文
- 外部环境反馈指导探索

**关键创新**：经典搜索算法与 LLM 能力融合。

**结果**：
- HumanEval 94.4%（GPT-4），83.8%（GPT-3.5）
- HotPotQA 71% 精确匹配
- WebShop 平均 75.9
- 显著优于 CoT、ReAct、ToT、Reflexion

**影响**：证明搜索算法可有效融合 LLM；LangGraph 与 LlamaIndex 提供实现。

### 2.7 语言 Agent 的认知架构（CoALA）

| 维度 | 说明 |
|---|---|
| **论文** | “Cognitive Architectures for Language Agents” |
| **会议** | TMLR 2024 |
| **作者** | Theodore R. Sumers, Shunyu Yao, Karthik Narasimhan, Thomas L. Griffiths |

**核心概念**

CoALA 提供一个**系统化框架**，从认知科学与符号 AI 传统出发理解 LLM Agent。

**三大组件**：
1. **模块化记忆**：工作记忆 + 长期记忆
2. **结构化动作空间**：内部动作 + 外部动作
3. **通用决策**：从简单提示到复杂规划

**关键创新**：提供统一分类法，连接现代 Agent 与认知架构研究传统。

**影响**：Agent 设计的关键参考框架。

## 3. 行业最佳实践（2025–2026）

### 3.1 Anthropic：“Building Effective Agents”

**核心理念**：从简单开始，多数应用不需要 Agent；优先优化单次 LLM 调用。需要复杂性时，优先工作流模式，再考虑自治 Agent。

**模式选择指南**：
| 模式 | 适用场景 |
|---|---|
| 提示链 | 固定步骤序列，每步需不同指令 |
| 路由 | 输入类型需完全不同处理 |
| 并行化 | 独立子任务或需要多视角投票 |
| 编排者-工作者 | 子任务不可预知的复杂任务 |
| 评估-优化 | 输出可评估并迭代提升 |
| 自治 Agent | 需要灵活多步决策的开放式任务 |

**关键原则**：
- 可预测任务优先用工作流
- 上下文工程 > 提示词工程
- 反模式：巨石型 Agent、过度规划、缺失可观测性
- 从直接 API 调用开始，不要先上框架

---

### 3.2 OpenAI：Agent 设计模式

**两种编排模式**：

1. **LLM 驱动编排（动态）**：
   - LLM 自主规划与选择 Agent/工具
   - 用清晰提示解释工具与运行参数
   - 通过循环与错误反馈实现自我改进
   - 优先专用 Agent，避免通用型

2. **代码驱动编排（确定性）**：
   - 结构化输出确保数据一致
   - 链式 Agent：研究 → 纲要 → 草稿 → 评审
   - 评估循环直到达标
   - 独立任务并行执行

**最佳实践**：
- 只在规则方法难以覆盖时使用 Agent
- 优先专精 Agent
- 加强提示词与评估系统
- 使用 handoff 清晰委派
- 在每个边界使用输入/输出护栏

---

### 3.3 Google：Agent 白皮书（76 页）

**模块化专精（Agent 微服务）**：
- 单个 Agent 责任过多会低效
- 设定 Parser、Critic、Dispatcher、Validator 等角色
- 模块化提升可测试性与可靠性
- 分布式系统降低瓶颈与调试成本

**八种多 Agent 模式（通过 ADK）**：
1. 顺序流水线
2. 并行执行
3. 循环 Agent（重试/迭代）
4. 层级委派
5. 人在回路
6. Agent-as-Tool
7. 动态路由
8. 协作讨论

**评估框架（三维）**：
1. **能力评估**：是否能完成设计能力
2. **轨迹分析**：工具使用是否正确高效
3. **响应评估**：输出是否正确完整

---

### 3.4 多 Agent 编排模式（跨行业）

| 模式 | 描述 | 适用场景 | 代价 |
|---|---|---|---|
| **顺序流水线** | Agent A → Agent B → Agent C | 依赖清晰 | 简单但延迟更高 |
| **并行/扇出** | 多 Agent 同时工作 | 独立子任务 | 更快但合并困难 |
| **层级/主管** | 管理者路由给专用工作者 | 分支复杂任务 | 灵活但主管成瓶颈 |
| **交接/委派** | 对话转给专家 | 自然任务切换 | 简单但拓扑受限 |
| **群聊** | Agent 协同讨论 | 创意/分析任务 | 丰富但难控 |
| **评估循环** | 生成 → 评估 → 精炼 | 质量要求高 | 更好但更慢 |

**新兴通信协议（2025）**：
- **MCP**：JSON-RPC 客户端/服务端工具调用标准（事实标准）
- **ACP**：RESTful HTTP 多段消息的 Agent 通信协议
- **A2A**：点对点任务委派，基于能力卡片
- **ANP**：基于 DID 的去中心化发现

## 4. 关键设计原则

### 4.1 工具使用模式

**函数调用架构**：
- LLM 生成结构化 JSON（函数名 + 参数）
- 应用校验调用并执行，返回结果
- LLM 处理结果并决定下一步或结束

**最佳实践**：
- 为每个工具提供清晰 schema 与自然语言说明
- 所有调用先校验（schema + 业务校验）
- 安全护栏：注入检测、动作空间限制、认证
- 分离工具发现（MCP）与工具执行
- 强制结构化输出保证调用有效

**模型上下文协议（MCP）——新兴标准**：
- 开源标准（“AI 的 USB-C”）
- 客户端/服务端架构：Host → Client → Server → 外部系统
- 定义 Resources（数据）、Tools（函数）、Prompts（模板）
- JSON-RPC + OAuth 2.0
- 规范版本 2025-11-25，Anthropic/OpenAI/Google/Microsoft 采纳
- 消除每个工具服务的定制集成成本

---

### 4.2 记忆架构

**三层模型（共识）**：

| 层级 | 范围 | 实现 | 类比 |
|---|---|---|---|
| **工作记忆** | 当前任务上下文 | 上下文窗口/对话缓冲 | RAM |
| **短期记忆** | 当前会话 | Session 状态/草稿 | L2 缓存 |
| **长期记忆** | 跨会话持久化 | 向量库/知识图谱/数据库 | 磁盘 |

**2025 创新**：
1. **统一记忆管理（AgeMem）**：将记忆操作暴露为工具，LLM 自主决定存取/更新/总结/丢弃；通过强化学习端到端优化。
2. **分层 OS 风格（MemoryOS）**：三层存储与 FIFO 更新，分段页组织与整合。
3. **连续记忆架构（CMA）**：从无状态 RAG 迈向内部状态维护，包含选择性保留、联想路由、时间链与抽象整合。
4. **记忆-推理协同（MEM1）**：联合优化记忆与推理，长任务下保持常量记忆，通过策略性丢弃达成。

**四项关键能力**：准确检索、测试时学习、长距离理解、选择性遗忘。

---

### 4.3 规划策略

| 策略 | 机制 | 优势 | 劣势 |
|---|---|---|---|
| **ReAct 循环** | 思考-行动-观察交替 | 简单有效，广泛支持 | 无前瞻，偏贪心 |
| **先计划后执行** | 先全局计划再执行 | 结构化、可预测 | 计划变更脆弱 |
| **自适应规划** | 计划 → 执行 → 复盘 | 容错强 | token 成本更高 |
| **树搜索（ToT/LATS）** | 多路径探索与回溯 | 战略任务最优 | 代价昂贵 |
| **层级规划** | 高层拆解子计划 | 复杂任务可控 | 协调开销 |
| **SOP 驱动（MetaGPT）** | 预定义流程 | 可预测、可验证 | 灵活性较弱 |

**实践建议（2025 共识）**：从 ReAct 开始；多步任务加入显式规划；仅在需要战略探索时使用树搜索。

---

### 4.4 错误恢复与自我修正

**挑战**：LLM 存在“自我修正盲点”——对自身错误纠正能力低，外部错误却更易修复（平均盲点率 64.5%）。

**有效模式**：
1. **检查点恢复（LangGraph）**：状态持久化，失败后回滚重试，避免重复执行
2. **语言反思（Reflexion）**：失败后反思并存入情节记忆
3. **外部验证环**：评审 Agent 检查输出，运行代码与校验
4. **自愈运行时（VIGIL）**：监督 Agent 监控日志，生成诊断与恢复方案
5. **元认知多 Agent 修正（MASC）**：重建执行检测错误，异常检测后修正

**缓解策略**：
- 使用最小 “Wait” 提示激活修正能力（盲点降低 89.3%）
- 优先外部反馈而非自反馈
- 护栏前置，避免错误传播
- 结构化输出校验作为第一道防线

---

### 4.5 上下文管理

**问题**：多步任务导致上下文充满历史、工具结果与中间推理，性能下降（主动干扰）。

**策略**：
1. **主动上下文管理（Sculptor）**：分片、摘要、隐藏/恢复，LLM 自主管理注意力
2. **上下文折叠（AgentFold）**：多尺度压缩，避免关键细节不可逆丢失
3. **记忆变换（InfiniteICL）**：将临时上下文转为长期参数更新
4. **实用方法**：
   - 对话历史滑动窗口 + 摘要
   - 工具结果截断与关键信息提取
   - 工作记忆与参考记忆分离
   - 长期知识使用 RAG 而非塞满上下文

---

### 4.6 护栏与安全

**分层防御框架（2025 企业最佳实践）**：

| 层级 | 控制项 |
|---|---|
| **身份与认证** | 唯一 Agent 身份、最小权限、短期凭据 |
| **输入护栏** | 注入检测、输入校验、内容过滤 |
| **执行隔离** | 沙箱、资源/时间限制、网络出口白名单 |
| **输出护栏** | Schema 校验、内容过滤、PII 检测、事实核验 |
| **人工监督** | 风险自适应审批、可审计干预 |
| **监控** | 红队、对抗测试、持续安全评估 |

**标准对齐**：NIST AI 风险管理框架、ISO/IEC 42001、OWASP LLM 应用 Top 10（v2025）。

**框架实现**：
- **OpenAI Agents SDK**：内建输入/输出护栏
- **CrewAI**：任务级护栏与人工介入回调
- **LangGraph**：基于中断的审批流与状态检查
- **Google ADK**：工作流 Agent 的验证环

### 4.7 可观测性与追踪

**五个关键指标**：
1. **成功率**（目标 >99%）
2. **响应延迟**（目标 p95 <3s）
3. **token 使用量**（成本监控）
4. **错误率**（工具失败、幻觉、护栏违规）
5. **业务结果**（任务完成、用户满意度）

**分布式追踪架构**：
- 追踪每次 LLM 调用、工具调用与决策点
- 使用一致 trace ID 进行跨 Agent 根因分析
- 捕获完整上下文：提示词、工具调用、检索数据与推理轨迹
- 从用户请求到最终响应的端到端埋点

**工具生态**：
- **LangSmith**：LangChain/LangGraph 原生，提供追踪与评估
- **Langfuse**：开源可观测性，支持 CrewAI/LangChain/自研 Agent
- **OpenTelemetry**：厂商无关标准
- **Prometheus + Grafana**：指标与可视化
- **Jaeger**：分布式追踪
- **Datadog、Braintrust**：企业级集成

**告警策略**：关注持续性问题（如 5 分钟内成功率 <95%），避免瞬时峰值告警疲劳。

### 4.8 评估框架

**双阶段评估策略**：

| 阶段 | 类型 | 目的 |

## 总结性建议

**生产应用**：优先 Anthropic 模式（最简单）。需要持久化与复杂编排选 LangGraph。需要快速多 Agent 原型选 CrewAI 或 OpenAI Agents SDK。  

**研究/实验**：分布式多 Agent 用 AutoGen，角色扮演研究用 CAMEL，软件工程流程用 MetaGPT。  

**企业部署**：Google ADK（Vertex AI）或 LangGraph（LangGraph Platform）部署方案最完整。  

**通用建议**：
- 先简单，后复杂
- 先评估，再提升复杂度
- 使用 MCP 统一工具集成
- 从第一天起就做可观测性
- 为失败而设计，检查点与恢复不可或缺
