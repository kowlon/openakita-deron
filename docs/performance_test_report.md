## 关键发现

### 1. LLM 调用占总耗时的 96%+

```
LLM 调用明细:
- 第一次: 24.2s (tokens=384→912)
- 第二次: 4.3s  (tokens=10238→68)   ← 决定调用工具
- 第三次: 9.9s  (tokens=10408→329)  ← 工具后处理
- 第四次: 9.7s  (tokens=724→425)    ← 最终输出

总 LLM 时间: 48.1s (96.2%)
非 LLM 时间: 1.9s (3.8%)
```

### 2. 工具执行后间隙极短

```
工具调用结束: 18:39:37.605
下一次 LLM 开始: 18:39:37.630
间隙: 25ms ✅ 非常快！
```

### 3. SSE 事件缺失导致"无反应"感觉

查看 SSE 输出序列:
```
tool_call_end → (无事件) → iteration_start → thinking_start
     ↑                                    ↑
     └── 这里用户看到工具完成 ─────────────┘ 然后等待 4-10s 才看到下一次 thinking
```

**问题**: 在 `tool_call_end` 和下一次 `thinking_start` 之间，没有任何中间状态通知用户。

## 优化建议

### 立即改进: 增加中间状态 SSE 事件

```python
# 在 reasoning_engine.py 工具执行后添加:
if progress_callback:
    await progress_callback("📋 正在分析工具结果...")

# 在下一次 LLM 调用前添加:
if progress_callback:
    await progress_callback("🧠 正在思考下一步...")
```

### 优化 LLM 调用次数

当前案例需要 4 次 LLM 调用:
1. Prompt 编译 (可以缓存)
2. 工具决策
3. 工具结果处理
4. 最终输出

优化方向:
- 缓存 Prompt 编译结果
- 合并工具决策和执行为一步
- 减少不必要的 LLM 轮次

### 前端体验优化

在工具调用结束后显示:
```
✅ list_directory 完成 (450 字符)
📋 正在分析结果...
```

而不是只显示:
```
✅ list_directory 完成 (450 字符)
[空白等待 4-10 秒]
💭 thinking...
```
