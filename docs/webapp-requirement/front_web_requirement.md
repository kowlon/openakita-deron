# 前端页面需求概述

本文档描述 SeeAgent Web UI 前端页面的功能需求和实现规范，强调会话可视化与步骤可控。

## 一、会话可视化与步骤可控

### 1. 核心目标
- 每次会话中的每一步都必须可视化展示
- 每一步都必须可控，可在执行后决定是否进入下一步

### 2. 执行模式

#### 2.1 Auto 模式（自动模式）
- 每一步执行完成后**自动进入**下一步
- 无需用户干预，适合所有任务

### 3. 可编辑结果与卡片化展示

**数据结构要求**：
- 步骤结果应为**数组格式**，便于卡片化展示
- 每个结果项包含：id、标题、内容、来源、选中状态

**卡片交互**：
- 每个结果项以卡片形式展示
- 卡片左侧显示勾选框
- 卡片右上角显示删除按钮
- 卡片内容支持点击编辑

**编辑后数据流**：
- 用户确认后，将**编辑后的结果**传递给下一步骤
- 后端需要支持接收用户修改后的中间结果

### 2. 左侧会话列表
- 展示每次会话的主题摘要
- 显示消息数量和最后更新时间
- 支持搜索会话
- 点击某个会话后加载该会话的聊天记录
- 点击"+"按钮创建新会话
- 支持删除会话

### 3. 中间主区域

#### 3.1 内容展示

**中间区域显示的内容**：

1. **用户消息** - 显示用户提出的问题（右侧对齐，蓝色主题）
2. **计时器** - 显示 TTFT 和总计时（紧随 AI 头像下方）
3. **步骤卡片** - AI 执行步骤以卡片形式展示（左侧对齐）
4. **文件卡片** - 生成的文件以卡片形式展示（新增）
5. **AI 总结** - 最终的文字回复

#### 3.2 计时器显示（新增）

##### TTFT（Time to First Token）
- **触发时机**：用户发送消息后立即开始
- **计时起点**：从 0 开始
- **停止时机**：收到第一个 token 后立即停止并锁定
- **显示规则**：停止后不再变化，后续步骤运行不影响 TTFT

##### 总计时
- **触发时机**：与 TTFT 同时开始
- **计时起点**：从 0 开始
- **停止时机**：所有输出完成后
- **显示规则**：运行中实时更新，完成后显示最终值

##### 计时器 UI 样式
```
运行中状态:
[●] TTFT: 1.25s | 总计: 1.25s
     ↑ 动画脉冲点

完成状态:
TTFT: 0.85s | 总计: 3.52s
```

#### 3.3 步骤卡片展示

**每个步骤卡片包含**：
- 状态图标（○ 等待中、⟳ 执行中、✓ 已完成、✕ 失败）
- 步骤类型图标（🧠 LLM调用、🔧 工具调用、⚡ 技能调用、💭 思考、📋 规划）
- 步骤标题
- 简要描述
- **步骤计时**（每个步骤独立计时）
- 右侧向下箭头按钮

**步骤计时规则**：
- 每个步骤有**独立的计时器**
- 步骤开始时从 **0 开始**计时
- 步骤执行中实时显示耗时（如 "1.25s"）
- 步骤完成/失败后停止计时，显示最终耗时
- 计时显示在步骤卡片右侧或标题旁

**步骤计时 UI 示例**：
```
执行中:
┌─────────────────────────────────────────┐
│  ⟳ 🔍 网络搜索              1.25s    ▼ │
│  正在搜索: 马斯克 最新新闻...            │

已完成:
┌─────────────────────────────────────────┐
│  ✓ 🔍 网络搜索              2.35s    ▼ │
│  找到 15 条相关新闻                      │
└─────────────────────────────────────────┘
```

**步骤状态表现**：
- 执行中：显示进度条和阶段说明，计时器实时更新
- 已完成：显示结果摘要和最终耗时
- 失败：显示错误信息和耗时

## 三、多轮对话（ChatGPT 风格）（新增）

### 1. 对话追加规则
- 新消息**追加**到对话列表末尾，不替换旧消息
- 每轮对话包含：用户问题 + 执行过程 + AI 响应

### 2. 历史对话显示
- 每轮历史对话显示：
  - 用户消息气泡
  - TTFT 和总计时（已完成的）
  - AI 响应摘要
- 不重复显示历史步骤卡片（减少视觉噪音）

### 3. 当前对话显示
- 用户消息气泡
- 实时计时器（运行中）
- 步骤卡片（仅核心步骤）
- 文件卡片（如有生成）
- AI 总结（完成后）

## 四、步骤过滤与聚合

### 1. 核心步骤的聚合与过滤显示

**需求说明**：步骤展示应进行智能聚合，只显示核心业务步骤，而非所有底层操作。

**示例场景**：

用户发送query：`"帮我查查曾德龙是谁，并帮我写入到pdf文件里"`

**期望显示的步骤（4个核心步骤）**：
```
步骤1: 🧠 意图分析
       识别到需要搜索"曾德龙"相关信息并输出为PDF

步骤2: 🔍 搜索
       正在搜索"曾德龙"相关信息...

步骤3: 📄 PDF输出
       正在生成PDF文件...

步骤4: ✅ 总结
       已完成搜索并输出PDF文件，保存路径：/xxx/曾德龙.pdf
```

### 2. 核心步骤（显示）

| 步骤类型 | 显示标题 | 匹配模式 |
|---------|---------|---------|
| 网络搜索 | 网络搜索 | web_search, search |
| PDF 生成 | PDF 文件生成 | 生成 .pdf 文件 |
| 文档生成 | 文档生成 | 生成 .docx/.xlsx 等 |
| 意图分析 | 意图分析 | thinking 类型 |

### 3. 内部步骤（隐藏）

**不应显示的底层操作**：
- Plan 管理（create_plan, update_plan 等）
- 文件读取（read_file）
- 命令执行（run_shell, bash）
- 技能信息获取（get_skill_info）
- 结果交付（deliver_artifacts）
- 总结/响应（总结, summary）
- 临时文件操作
- 配置加载

**聚合原则**：将多个原子操作合并为一个有意义的业务步骤，让用户看到的是"做了什么"，而不是"怎么做的"。

### 4. 简单 Q&A 处理
- 如果没有工具调用，直接显示 AI 文本回复
- 不显示"意图分析"等步骤卡片
- 计时器正常显示

## 六、用户体验增强

### 1. 左侧会话摘要自动生成
- 当用户发送query并收到AI响应后，**自动生成会话主题摘要**
- 摘要应简洁概括用户的query意图（建议10-20字）
- 生成时机：第一个步骤完成或收到AI响应后
- 示例：用户发送"帮我查查曾德龙是谁，并帮我写入到pdf文件里"，摘要应为"查询曾德龙信息并输出PDF"

### 2. 原始用户query的正确展示
- 中间对话区域必须**准确显示用户发送的原始query**
- 不得将用户的中文query替换为英文或其他翻译版本
- 用户消息应实时更新：发送消息后立即在界面上显示

### 3. 欢迎页面
当没有会话时显示欢迎页面（ChatGPT 风格）：
- Logo 和标题
- 快捷操作卡片（网络搜索、文档处理、代码助手、数据分析）
- 输入框

## 七、数据结构

### 1. 会话（Session）
```typescript
interface Session {
  id: string
  title: string
  timestamp: number
  status: 'active' | 'completed' | 'paused'
  conversationHistory: ConversationTurn[]
}
```

### 2. 对话轮次（ConversationTurn）
```typescript
interface ConversationTurn {
  id: string
  userMessage: string
  steps: Step[]           // 执行步骤
  summary: string | null  // AI 最终回复
  artifacts: Artifact[]   // 生成的文件（新增）
  timestamp: number
  // 计时信息
  startTime: number       // 消息发送时间
  firstTokenTime: number  // 首 token 时间（TTFT 锁定点）
  endTime: number         // 完成时间
}
```

### 3. 文件产物（Artifact）（新增）
```typescript
interface Artifact {
  id: string
  type: 'pdf' | 'word' | 'excel' | 'image' | 'other'
  filename: string
  filepath: string        // 服务器路径
  size?: number          // 文件大小（字节）
  downloadUrl?: string   // 下载链接
}
```

### 4. 步骤（Step）
```typescript
interface Step {
  id: string
  type: 'llm' | 'tool' | 'skill' | 'thinking' | 'planning'
  status: 'pending' | 'running' | 'completed' | 'failed'

  summary: string
  startTime: number
  endTime?: number
  duration?: number
  input?: Record<string, unknown>
  output?: string
  category: 'core' | 'internal'
}
```

## 八、后端事件处理

### 1. SSE 事件类型
| 事件类型 | 用途 |
|---------|------|
| `thinking_start` | 思考开始（不显示步骤卡片） |
| `tool_call_start` | 工具调用开始 |
| `tool_call_end` | 工具调用结束 |
| `text_delta` | 文本输出（用于检测首 token） |
| `done` | 任务完成 |
| `artifact_created` | 文件生成（新增） |

### 2. TTFT 检测逻辑
```
1. 用户发送消息 → 记录 startTime
2. 收到 text_delta 事件 → 记录 firstTokenTime（仅第一次）
3. TTFT = firstTokenTime - startTime
4. TTFT 显示后不再变化（锁定）
```

### 3. 步骤计时逻辑
```
1. 收到 tool_call_start 事件 → 创建步骤，记录 stepStartTime
2. 步骤计时器实时显示: Date.now() - stepStartTime
3. 收到 tool_call_end 事件 → 记录 stepEndTime
4. 步骤最终耗时 = stepEndTime - stepStartTime
5. 步骤完成后计时器停止，显示最终耗时
```

### 4. 三种计时器对比

| 计时器 | 开始时机 | 停止时机 | 作用域 |
|-------|---------|---------|-------|
| TTFT | 消息发送 | 首 token 到达 | 整个对话轮次 |
| 总计时 | 消息发送 | 所有输出完成 | 整个对话轮次 |
| 步骤计时 | 步骤开始 | 步骤完成 | 单个步骤 |

### 3. 步骤过滤逻辑
```typescript
function shouldShowStep(step: Step): boolean {
  // 1. 如果是 internal 类别，隐藏
  if (step.category === 'internal') return false

  // 2. 检查是否匹配内部步骤模式
  if (INTERNAL_STEP_PATTERNS.some(p => p.test(step.title))) return false

  // 3. 检查是否匹配核心步骤模式
  if (CORE_STEP_PATTERNS.some(p => p.test(step.title))) return true

  // 4. 默认隐藏
  return false
}
```

## 九、待修复的 Bug

### 1. TTFT 计时器问题（P0）
**问题描述**: TTFT 在首 token 输出后没有停止，后续步骤运行时继续增加

**修复方案**:
- TTFT 在收到 `text_delta` 首次事件后立即锁定值
- 后续步骤运行不影响 TTFT 显示
- TTFT 显示区域应该只显示一次，不随步骤增加

### 2. 中间步骤不显示问题（P0）
**问题描述**: 复杂任务的执行步骤没有正确显示

**修复方案**:
- 检查 `useChat.ts` 中的步骤过滤逻辑
- 确保 `tool_call_start` 事件正确创建步骤
- 确保核心步骤的 `category` 正确设置为 `core`
- 简单 Q&A 不应显示步骤卡片
