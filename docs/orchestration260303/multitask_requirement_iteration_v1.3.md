# 多任务编排迭代需求文档（相对 v1.2）

> 基线文档：`multitask_requirement_structured.md`  
> 版本：v1.3（迭代）  
> 日期：2026-03-06  
> 状态：需求补充与重构范围确认

---

## 1. 文档目标与适用范围

本迭代需求用于在现有多任务编排能力基础上，补足以下核心能力：

1. 从“规则匹配路由”升级为“LLM 上下文路由”
2. 支持“多会话任务激活管理”，明确任务焦点切换语义
3. 完善聊天区任务卡片交互，支持“首卡大卡片 + 后续简洁卡片”
4. 实现路由与场景触发 Prompt 独立管理，降低维护成本

本迭代文档仅定义增量需求，不重复覆盖基线文档已稳定内容。

---

## 2. 相对 v1.2 的核心变化

### 2.1 路由机制变化

- **原机制**：有活跃任务时默认转发当前步骤；无活跃任务时规则匹配场景
- **新机制**：统一由 LLM 基于上下文决策：
  - 是否进入任务模式
  - 路由到哪个任务
  - 路由到任务内哪个步骤
  - 是否触发新最佳实践场景

### 2.2 任务激活机制变化

- **原机制**：每个会话只有一个映射中的活跃任务，缺少显式“失焦/激活”状态
- **新机制**：引入会话级与全局级任务激活管理，支持抢占、显式激活、恢复

### 2.3 前端任务卡片机制变化

- **原机制**：仅展示当前活跃任务卡片，展示策略不区分首轮和后续
- **新机制**：任务相关对话全部卡片化展示：
  - 第一次展示大卡片（信息完整）
  - 后续展示简洁卡片（轻量状态）

### 2.4 Prompt 管理变化

- **原机制**：路由与场景触发依赖规则逻辑，SubAgent 步骤 Prompt 内嵌在场景 YAML
- **新机制**：
  - MainAgent 到 SubAgent 路由判定 Prompt 独立维护
  - 最佳实践触发判定 Prompt 独立维护
  - SubAgent 本身步骤 Prompt 继续保留在对应场景配置文件中（不做独立化改造）

## 3. 迭代业务规则

### 3.1 单用户活跃焦点规则

- 系统同一时刻仅允许一个“全局焦点任务”（用于聊天区任务态路由）
- 同一会话内可存在多个任务，但仅一个“会话激活任务”
- 新触发的最佳实践任务可抢占当前会话焦点，旧任务进入“非激活态（出域）”

### 3.2 会话内显式激活规则

- 用户点击任务卡片时，系统将该任务设为“显式激活”
- 用户通过自然语言表达“切回某任务”时，LLM 可判定并触发激活切换
- 显式激活优先级高于自动路由推荐

### 3.3 任务路由优先级规则

1. 用户显式激活目标（卡片点击/明确指令）
2. LLM 判定当前消息属于某个已存在任务
3. LLM 判定应新建最佳实践任务
4. 普通对话

### 3.4 任务内步骤路由规则

- 活跃任务内由 LLM 判定目标步骤
- 若目标步骤对应 SubAgent 未激活，则先激活再执行
- 若输入条件不满足，返回 AskUser 交互，不直接执行步骤

## 4. 任务状态与激活状态

### 4.1 生命周期状态（保留）

- PENDING
- RUNNING
- WAITING_USER
- COMPLETED
- CANCELLED
- FAILED

### 4.2 新增激活状态（迭代）

用于表达“任务是否为当前焦点”，与生命周期正交：

- ACTIVE：当前会话焦点任务
- EXPLICIT_ACTIVE：用户显式激活任务
- OUT_OF_FOCUS：被新任务抢占后暂时出域
- INACTIVE：会话内存在但未激活

## 5. 用户交互需求（增量）

### 5.1 聊天区任务卡片展示

- 任务首次进入当前会话时显示大卡片：
  - 场景名、任务状态、进度、当前步骤、可操作按钮
- 同任务后续轮次显示简洁卡片：
  - 状态、进度、当前步骤摘要

### 5.2 卡片交互

- 点击卡片：激活该任务
- 卡片内支持：
  - 确认步骤
  - 切换步骤
  - 取消任务

### 5.3 卡片展示边界

- 普通聊天不展示任务卡片
- 任务相关对话必须展示任务卡片（首轮大卡片，后续简洁卡片）

---

## 6. 后端能力需求（增量）

### 6.1 多会话任务激活管理能力

- 提供任务激活切换能力
- 提供会话当前激活任务查询能力
- 提供任务被抢占/出域状态管理能力

### 6.2 LLM 路由决策能力

- 替换规则匹配逻辑：
  - 场景触发判定
  - 任务选择判定
  - 步骤选择判定
  - 输入满足性判定
- Prompt 管理范围限定为：
  - 路由判定 Prompt（MainAgent -> SubAgent）
  - 最佳实践触发 Prompt
  - SubAgent 步骤 Prompt 仍由场景配置文件维护

### 6.3 事件通知能力

后端需输出结构化事件供前端驱动卡片展示与状态同步：

- `task_card_created`
- `task_card_compact`
- `task_activation_changed`
- `task_out_of_focus`

---

## 7. 前端能力需求（增量）

### 7.1 会话维度任务视图

- 任务列表按会话隔离
- 当前会话仅展示本会话任务卡片
- 会话切换时同步刷新该会话激活任务

### 7.2 激活操作联动

- 卡片点击后触发激活接口
- 收到激活变更事件后实时更新 UI
- 对“被抢占任务”提供可见提示

### 7.3 卡片形态管理

- 同任务 ID 在会话内首次渲染为大卡片
- 后续渲染自动降级为简洁卡片

## 8. 非功能需求

### 8.1 可维护性

- 路由判定与场景触发 Prompt 独立管理，支持非代码人员维护
- SubAgent 步骤 Prompt 保持配置文件维护，避免本迭代扩大改造范围
- 决策逻辑与执行逻辑解耦，降低模块耦合

### 8.2 可观测性

- 记录每次路由决策的置信度、目标任务、目标步骤、决策理由
- 记录任务激活切换链路，支持问题回溯

### 8.3 兼容性

- 保留旧规则匹配作为降级开关（灰度期间）
- 保持现有任务创建、确认、取消主流程可用

---

## 9. 验收标准

### 9.1 业务验收

- 在已有活跃任务时触发新最佳实践，系统可正确抢占并将旧任务出域
- 用户点击历史任务卡片后，该任务可立即成为激活任务并接管后续任务对话
- 同任务在聊天区实现“首大后简”的卡片展示

### 9.2 技术验收

- 路由主路径不再依赖规则关键词匹配
- 场景触发与步骤路由均有 LLM 决策记录
- 前后端对任务激活状态一致

---

## 10. 迁移与上线要求

### 10.1 灰度策略

1. 先启用任务激活管理器（不切主路由）
2. 再启用场景触发 LLM 判定
3. 最后启用步骤路由 LLM 判定并关闭规则主路径

### 10.2 回滚策略

- 任一阶段可切回规则路由
- 激活管理可回退为单映射会话任务模型

---

## 11. 术语补充

- **任务激活**：任务被选为当前会话路由目标的状态
- **出域**：任务仍存在但不再是当前焦点任务
- **显式激活**：由用户主动操作触发的激活，不依赖自动判定
- **任务相关对话**：被判定为针对任务/步骤的用户消息
