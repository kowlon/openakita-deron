# 多任务简洁设计描述

> 日期：2026-03-04  
> 范围：基于《多任务编排需求整理》《OpenAkita 多任务编排技术设计文档》的简洁对齐版

---

## 1. 总体目标

- 建立一套多任务编排机制：识别最佳实践场景 → 拆解为多步骤任务 → 逐步执行 → 用户可编辑与确认。
- 提供前端可视化流程与交互：让用户理解当前在做什么、处于哪一步、可控推进/回退/取消。

## 2. 核心概念

### 2.1 核心角色

- MainAgent：对话入口与路由中心；负责场景匹配、任务创建、消息路由与任务态管理。
- SubAgent：步骤执行代理；独立进程、独立对话历史；在“步骤模式”中承载每个 step 的交互与执行。
- TaskSession：任务会话；管理任务生命周期、步骤会话、上下文传递、步骤切换与确认。

### 2.2 SubAgent 与 WorkerAgent 的统一关系

- SubAgent 与 WorkerAgent 统一为“独立进程 Agent”架构，差异通过配置体现（执行语义、生命周期、Brain 模式、工具可见集合、系统提示词等）。
- 复用 WorkerAgent 的进程与 ZMQ 通信方式，以“步骤执行模式”落地 SubAgent。

## 3. 入口与用户流程

### 3.1 触发入口

- 对话触发：MainAgent 接收用户请求 → 场景匹配命中最佳实践 → 创建任务 → 引导用户确认 → 进入步骤执行。
- 最佳实践入口：用户点击最佳实践直接创建任务 → 右侧显示任务区 → 引导用户输入第一步依赖信息。

### 3.2 消息路由（MainAgent 作为路由中心）

- 有活跃任务且处于 step 模式：将用户消息路由到当前步骤 SubAgent。
- 无活跃任务：先尝试场景匹配；匹配成功则创建任务并进入第一步；否则按普通对话处理。

## 4. 步骤执行机制

### 4.1 独立对话与快照

- 每个步骤拥有独立的 SubAgent 进程与对话历史。
- TaskSession 保留每一步 StepSession（messages/output/status/sub_agent_id 等）用于追溯与前端展示。

### 4.2 上下文传递（自动注入）

- 上一步在“用户确认”后生成结构化输出并写入 TaskSession.context（通常以 output_key 标识）。
- 下一步在构建 system_prompt 时自动注入“前置步骤输出”，让 SubAgent 基于已确认的信息继续执行。

### 4.3 用户确认与编辑

- 支持 Edit 模式：步骤输出可由用户编辑并确认，确认后的版本作为下一步输入。
- 用户显式执行“确认/下一步”触发 complete_step，随后推进到下一步骤。

### 4.4 步骤切换与取消

- 用户可显式切换到任意步骤；切换前检查步骤条件（不满足则提示）。
- 支持取消任务：销毁所有 SubAgent 进程并清理任务状态。

## 5. 工具与能力边界

### 5.1 工具可见集合裁剪

- tools/mcp_tools：作为可执行工具集合，决定对模型“可见且可调用”的工具范围。
- skills：仅注入提示词（能力/边界约束），并通过技能工具调用串联，不作为默认任意可执行工具集合。

### 5.2 超出能力范围的处理策略

- SubAgent 识别当前请求超出工具边界时，必须明确告知边界，并提供选项：
  - 借用其他工具（需要用户确认）
  - 切换到其他步骤
  - 切换到自由模式（使用 MainAgent 全量能力）

## 6. 状态模型与生命周期

### 6.1 任务状态

- PENDING：等待开始
- RUNNING：执行中
- WAITING_USER：等待用户确认/编辑
- COMPLETED：完成
- CANCELLED：已取消

### 6.2 生命周期约束

- 任务完成后销毁所有 SubAgent。
- 任务取消后清理所有状态与进程资源。
- 上一步输出必须可追溯，并在确认后作为下一步输入来源。

## 7. 前端展示要点（与现有 seeagent 布局一致）

- 左侧：对话列表上方新增“最佳实践任务列表”（最多 5 行），提供“更多”入口。
- 中间：主对话区保持现有对话方式，新增“任务卡片”（展示简述、步骤/子代理信息、确认入口）。
- 右侧：
  - 步骤卡片：展示步骤输出内容
  - 任务卡片：上部展示多个 SubAgent 名字/说明/状态；下部仅展示最后一步输出，支持编辑保存
  - 步骤总结不在右侧展示，统一在主对话框展示并标注步骤标识

## 8. 技术落地摘要（关键实现点）

- SubAgent 以独立进程运行，内部创建完整 Agent 实例，使用 ReasoningEngine 保证推理/工具能力与 MainAgent 一致。
- TaskSession 负责：
  - dispatch_step：向当前步骤 SubAgent 发送请求（ZMQ）
  - complete_step：用户确认后生成输出并写入上下文
  - switch_to_step：显式切换步骤（含条件校验）
  - mode：step/free 模式切换
- MessageRouter 负责统一路由入口，按“活跃任务/场景匹配/普通对话”分流。
